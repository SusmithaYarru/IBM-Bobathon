ACE + Agent Processing Pipeline

1

HTTP Input — raw HL7 ingestion HTTP

ACE listens on /hl7/ingest (port 7080). The body is parsed using the BLOB domain — ACE does not need to understand HL7 syntax at this stage, it treats it as raw bytes.

2

ESQL HL7Parser — tokeniser ESQL

Pure ESQL splits the pipe-delimited message into a LocalEnvironment.HL7 element tree. Each segment (MSH, PID, OBX…) is indexed with fields and ^-split components. No external JAR required.

3

Java HL7SegmentParser — structured parser Java

The MbJavaComputeNode subclass builds a typed MbElement tree, handles multi-occurrence segments (e.g. multiple OBX), and produces a compact JSON summary string for the AI agent.

4

ESQL HL7Validator — required field checks ESQL

Validates MSH-3, MSH-9, MSH-10, MSH-11, MSH-12, PID-3, PID-5 and more. Writes a structured validation report into LocalEnvironment. Routes to the error queue if isValid = false.

5

AI Agent Pipeline — 4 agents watsonx.ai

ACE's HTTP Request node POSTs the JSON summary to the Node.js agent on :4000/process. Four agents run: Classify → Validate → Enrich → Summarise. Returns an AgentResult that drives routing and ACK code.

6

MQ Routing + ACK Builder MQ ESQL

Messages are routed to HL7.ADT.IN, HL7.ORU.IN, or HL7.ORM.IN MQ queues by message type. ESQL constructs a standards-compliant MSH+MSA ACK (AA or AE) returned to the sender.

Four AI Agents

C ClassifierAgent

  • Identifies message type & trigger event
  • Assigns urgency: critical / high / medium / low
  • Scans OBX flags: LL HH AA → escalates
  • Determines routing target queue
  • LLM generates one-sentence reason

V ValidatorAgent

  • Tier 1: deterministic required-field rules
  • MSH-3/9/10/11/12, PID-3/5 checks
  • PID-7 date format, PID-8 sex code
  • Tier 2: AI-assisted ICD-10 / OBX unit checks
  • Returns error/warning issues array

E EnrichmentAgent

  • FHIR patient demographics lookup (by MRN)
  • Prior encounter history (last 3)
  • Drug allergy cross-check for ORM orders
  • ReAct-style tool loop: call → observe → stop
  • Stub FHIR store for offline demo

S SummaryAgent

  • Aggregates all prior agent results
  • Stub: deterministic template engine
  • Live: IBM Granite-13b clinical prompt
  • Outputs human-readable event narrative
  • Includes allergy alerts + critical flags

Key Design Decisions

  • 1
    No external HL7 library inside ACE Pure ESQL tokenisation keeps the BAR file lightweight. No unsupported third-party JARs inside the integration server. The Java Compute node uses only core ACE MbJavaComputeNode APIs.
  • 2
    AI layer runs as a sidecar — outside ACE The Node.js agent communicates with ACE over a simple HTTP POST. ACE flows stay deterministic and independently deployable. The AI logic is testable with plain Jest without any ACE runtime.
  • 3
    LangChain-style tool pattern without the framework Each agent registers tool objects with name, description, parameters, and execute(). This mirrors the LangChain pattern but requires zero additional dependencies — easy to replace with real LangChain later.
  • 4
    Offline / stub mode for demo resilience Set WATSONX_STUB=true in .env. Every agent returns realistic deterministic responses without any API calls. The full pipeline — classify → validate → enrich → summarise — runs completely offline.
  • 5
    Two-tier validation: rules first, AI second Deterministic required-field checks always run (fast, 100% reliable). AI validation is called only after tier-1 passes, avoiding noisy LLM output on structurally broken messages.
